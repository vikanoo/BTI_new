# Требования к проекту: Телеграм-бот анализа перепланировки БТИ

## 1. Общее описание

**Продукт:** Телеграм-бот с бэкэндом на N8N workflows
**Цель:** Помочь пользователям сравнить официальный план БТИ с реальной планировкой квартиры и определить наличие и законность перепланировки

---

## 2. Участники системы

| Роль | Описание |
|------|----------|
| Пользователь | Владелец/арендатор квартиры, желающий проверить перепланировку |
| Менеджер | Эксперт-человек, получающий уведомление в случае низкой уверенности агента в результатах анализа; проводит анализ вручную вместо AI-агента |

---

## 3. Функциональные требования

### 3.1 Получение плана БТИ

- Пользователь загружает план БТИ в виде:
  - Фотографии (JPEG, PNG)
  - Файла (PDF)
- Если загружен PDF: N8N передаёт файл в Python-микросервис для конвертации первой страницы в PNG (`pdf2image`), результат сохраняется в Supabase Storage
- Бот подтверждает успешное получение и уведомляет о начале обработки

### 3.2 Анализ плана и сегментация по комнатам

- Gemini 2.0 Flash анализирует изображение плана БТИ и возвращает JSON сегментации:
  ```json
  {
    "rooms": [
      {
        "id": "room_1",
        "name": "Жилая комната 1",
        "region_percent": { "x1": 0.05, "y1": 0.10, "x2": 0.45, "y2": 0.55 }
      },
      {
        "id": "kitchen",
        "name": "Кухня",
        "region_percent": { "x1": 0.50, "y1": 0.10, "x2": 0.95, "y2": 0.55 }
      }
    ]
  }
  ```
- Поддерживаемые типы помещений: жилые комнаты, кухня, санузел(ы), коридор, балкон/лоджия, кладовая
- JSON сегментации сохраняется в Supabase (таблица `sessions`) — используется в шагах 3.3 и 3.7
- Python-сервис использует `region_percent` для нанесения подписей; размеченное изображение отправляется пользователю

### 3.3 Разметка плана и инструкции по съёмке

- Gemini 2.0 Flash анализирует план и возвращает структурированный JSON с точками съёмки:
  ```json
  {
    "rooms": [
      {
        "id": "room_1",
        "name": "Жилая комната 1",
        "shots": [
          {
            "shot_id": "room_1_shot_1",
            "position": "северо-западный угол",
            "direction": "на юго-восток",
            "instruction": "Встаньте в угол, снимайте всю комнату целиком",
            "x_percent": 0.15,
            "y_percent": 0.22
          }
        ]
      }
    ]
  }
  ```
- Координаты `x_percent` / `y_percent` — нормализованные (0–1) относительно размеров изображения плана; Python-сервис использует их для нанесения точек
- Для каждой точки формируются инструкции: позиция, направление, угол охвата, высота, освещение
- Цель — обеспечить максимальное покрытие каждого помещения для точного анализа

### 3.4 Сбор фотографий реальной планировки

- Бот последовательно запрашивает фотографии по каждой точке съёмки
- Отображает пользователю:
  - Миниатюру разметки с выделенной текущей точкой
  - Текстовую инструкцию для данной точки
- После получения фото — подтверждение и переход к следующей точке
- Возможность перефотографировать, если пользователь не уверен в качестве снимка
- **Важно:** бот явно просит отправлять фото **как файл (документ)**, не как обычное фото — Telegram сжимает photo-тип до ~5 МБ и снижает разрешение, что ухудшает точность анализа GPT-4o

**Отслеживание статусов точек съёмки** (хранится в Supabase, таблица `sessions`, поле `shots_status`):

| Статус | Значение |
|--------|----------|
| `pending` | Точка ещё не запрошена |
| `requested` | Бот отправил инструкцию, ожидает фото от пользователя |
| `received` | Фото получено и сохранено в Supabase Storage |

N8N читает этот статус при каждом входящем сообщении и определяет, какую точку запрашивать следующей — без обращения к LLM. Паттерн взят из предыдущей версии воркфлоу (там хранился в Google Sheets); в новой реализации переведён в Supabase.

### 3.5 Итоговый анализ перепланировки

- **Модель:** GPT-4o (`gpt-4o`) в режиме `detail:high` — обеспечивает точный анализ мелких деталей на фотографиях
- Если уверенность агента в результатах анализа недостаточна (например, низкое качество фото, неоднозначные изменения), N8N отправляет уведомление менеджеру; менеджер проводит анализ вручную вместо AI-агента
- Агент сопоставляет план БТИ с полученными фотографиями
- Выявляет расхождения:
  - Снос/возведение стен
  - Перенос дверных/оконных проёмов
  - Изменение площади помещений
  - Объединение/разделение комнат
  - Перенос санузла или кухни
- Формирует подробный отчёт по каждому выявленному отклонению

### 3.6 Оценка законности перепланировки

- Агент обращается к базе знаний (RAG по нормативам) для классификации каждого выявленного изменения
- Каждое изменение классифицируется по категориям:
  - ✅ Согласуемая перепланировка (законная, не требует действий)
  - ⚠️ Требующая согласования (нужно оформить документы)
  - ❌ Незаконная / опасная (затрагивает несущие конструкции, мокрые зоны, эвакуационные пути)
- Итоговое резюме формируется как **список классифицированных изменений**, а не как единый процент — процент может вводить в заблуждение без понимания методологии
- Формируются рекомендации по легализации для каждого пункта

### 3.7 Визуализация перепланировки

- GPT-4o (шаг 3.5/3.6) возвращает JSON с описанием локаций изменений:
  ```json
  {
    "changes": [
      {
        "room_id": "room_1",
        "type": "wall_removed",
        "classification": "illegal",
        "location_description": "стена между комнатой 1 и кухней"
      }
    ]
  }
  ```
- Python-сервис сопоставляет `room_id` с данными сегментации (шаг 3.2) и наносит подсветку на план
- Логика визуализации: **фон плана нейтральный**, подсвечиваются только зоны изменений:
  - Жёлтым — изменения, требующие согласования
  - Красным — незаконные / опасные изменения
  - Без изменений — план остаётся как есть (не закрашивается зелёным)
- Итоговое аннотированное изображение + нумерованные подписи отправляются пользователю
- Опционально (Phase 2): PDF-отчёт с планом и полным описанием

> ⚠️ Ограничение точности: GPT-4o не возвращает пиксельные координаты — локация изменений определяется через привязку к room_id из сегментации, а не через геометрические bounding-box'ы. Точность нанесения разметки соответствует точности сегментации плана.

## 4. База знаний (правила перепланировки)

### 4.1 Хранение правил

- PDF-файлы с нормативными актами загружаются один раз администратором
- Правила нарезаются по смысловым блокам (по статьям/пунктам, не по страницам) и векторизуются через `text-embedding-3-small`
- Embeddings хранятся в **Supabase pgvector**
- Агент использует RAG для точечного поиска релевантных норм — **без необходимости читать все PDF при каждом запросе**
- База знаний обновляется администратором при изменении нормативов

### 4.2 Источники правил

- Жилищный кодекс РФ (ст. 25-29)
- СНиП и ГОСТ (санитарные, строительные нормы)
- Региональные нормативы (при необходимости)
- Внутренние правила и прецеденты (custom)

## 5. Архитектура системы

```
Пользователь (Telegram)
        │
        ▼
  Telegram Bot API
        │
        ▼
   N8N Workflow (оркестратор)
   ├── Webhook-триггер (входящие сообщения)
   ├── Чтение/запись состояния диалога → Supabase sessions
   ├── Vision LLM: анализ плана, генерация точек → Gemini 2.0 Flash
   ├── Vision LLM: финальный анализ перепланировки → GPT-4o
   ├── RAG-запросы (embeddings) → Supabase pgvector
   ├── Аннотация изображений → Python-микросервис (Flask + PIL)
   └── Отправка ответов → Telegram Bot API
        │
   ┌────┴──────────────────────────────┐
   │                │                  │
Supabase        Supabase           Python
pgvector        Storage          микросервис
(нормативы)  (планы и фото)   (разметка плана)
```

### 5.1 Структура N8N Workflow (входящая маршрутизация)

Каждое входящее сообщение от пользователя обрабатывается единым Webhook-триггером. Первый узел — **Switch** — определяет тип сообщения и направляет поток в нужную ветку:

```
Telegram Webhook Trigger
        │
        ▼
   Supabase: проверка chat_id в allowed_users
   ├── не найден → ответ "Доступ запрещён" → стоп
   └── найден ↓
        │
        ▼
   Switch (тип сообщения)
   ├── document/photo → ветка приёма файла (план БТИ или фото комнаты)
   ├── text "/start"  → ветка приветствия и инициализации сессии
   ├── text           → ветка текстового ввода (подтверждения, команды)
   └── callback_query → ветка обработки inline-кнопок (повтор съёмки и др.)
        │
        ▼
   Supabase: чтение sessions (chat_id → текущий шаг, статусы точек)
        │
        ▼
   Роутинг по текущему шагу сессии → нужный workflow-блок
```

Паттерн Switch по типу сообщения взят из предыдущей версии воркфлоу — он корректно обрабатывает все типы входящих данных Telegram без дополнительных проверок в каждой ветке.

### 5.2 Компоненты

| Компонент | Роль | Технология |
|-----------|------|------------|
| Telegram Bot | Интерфейс пользователя | Telegram Bot API |
| N8N | Оркестрация workflow, управление диалогом | N8N (self-hosted) |
| Supabase (таблица sessions) | Хранение состояния диалога (chat_id → текущий шаг) | Supabase PostgreSQL |
| Gemini 2.0 Flash | Анализ плана БТИ, сегментация, генерация точек съёмки | Google Gemini API |

| text-embedding-3-small | Векторизация нормативных правил | OpenAI API |
| Supabase pgvector | Хранение embeddings правил (RAG) | Supabase |
| Supabase Storage | Планы БТИ и фотографии пользователей | Supabase |
| Python-микросервис (Flask) | Аннотация изображений: PIL/Pillow для нанесения разметки | Python + PIL; деплой на том же VPS что N8N (Docker Compose) |

## 6. Пользовательский сценарий (User Flow)

```
1. /start → Приветствие + инструкция                          [N8N]
2. Загрузка плана БТИ → Подтверждение получения               [N8N]
3. Анализ плана → Размеченное изображение с комнатами         [Gemini 2.0 Flash]
4. Генерация точек съёмки → Инструкции по фотографированию    [Gemini 2.0 Flash]
5. Последовательный сбор фото (комната за комнатой)           [N8N — без LLM]
   - N8N отслеживает состояние: текущая комната, кол-во фото
   - Запросы, подтверждения, повтор съёмки — чистая логика N8N
6. Финальный анализ перепланировки                            [GPT-4o + RAG]
   - RAG-запрос к Supabase pgvector для получения релевантных норм
   - GPT-4o сравнивает план БТИ + все фото + нормы
7. Итоговый отчёт:                                            [N8N + Python-сервис]
   - Список изменений с классификацией (✅ / ⚠️ / ❌)
   - Рекомендации по легализации для каждого пункта
   - Аннотированный план (цветовая разметка через Python PIL)
```

## 9. Ограничения и допущения

### Точность анализа
- Анализ носит **информационный характер** и не является юридически значимым заключением
- Система не заменяет официальную техническую экспертизу БТИ или лицензированного инженера
- **Принципиальное ограничение:** сравнение 2D плана с 3D перспективными фотографиями не позволяет выявлять мелкие изменения (перенос проёма на 20–50 см, незначительное изменение площади). Система надёжно определяет только **очевидные изменения**: снос/возведение стен, объединение/разделение комнат, появление/устранение дверных проёмов
- Точность анализа критически зависит от качества и полноты предоставленных фотографий

### Данные и приватность
- Фотографии квартиры передаются в OpenAI API и Google Gemini API для обработки — пользователь должен быть уведомлён об этом
- Обработка персональных данных регулируется **152-ФЗ**; необходимо пользовательское соглашение
- Фотографии хранятся в Supabase Storage; необходимо определить срок хранения и политику удаления

### Технические ограничения
- Telegram ограничивает размер загружаемого файла до 20 МБ
- При прерывании диалога пользователем состояние сохраняется в Supabase; сессия активна 48 часов, затем сбрасывается

## 10. Этапы разработки (MVP → Full)

### MVP (Phase 1)

#### 10.1 Создание нового N8N Workflow с нуля

- [ ] **Настройка Supabase:** создать таблицы `sessions`, `shots`, `embeddings`, `allowed_users`; подключить расширение pgvector; создать Storage bucket для планов и фото
- [ ] **Whitelist:** создать таблицу `allowed_users (chat_id bigint primary key)` в Supabase; первый узел после триггера — проверка `chat_id` в этой таблице; если не найден — бот отвечает отказом и прерывает workflow
- [ ] **Каркас воркфлоу:** Telegram Webhook Trigger → проверка whitelist → Switch (document / photo / text / callback_query) → чтение `sessions` из Supabase → роутинг по текущему шагу
  - *Взять из старого воркфлоу:* структуру Switch-узла по типу Telegram-сообщения
- [ ] **Шаг 1 — Приём плана БТИ:** ветка `document/photo` → определение типа файла → если PDF: конвертация в PNG (Code Node / HTTP к Python-сервису) → сохранение в Supabase Storage → обновление `sessions.step = 'plan_received'`
- [ ] **Шаг 2 — Анализ плана и сегментация:** Gemini 2.0 Flash → JSON с комнатами → сохранение в `sessions.rooms_json` → отправка размеченного изображения пользователю (Python-сервис)
- [ ] **Шаг 3 — Генерация точек съёмки:** Gemini 2.0 Flash → JSON с точками → сохранение в `sessions.shots_json` → инициализация статусов `shots_status` (все `pending`) → отправка первой инструкции
- [ ] **Шаг 4 — Последовательный сбор фото (без LLM):** ветка `document/photo` при `step = 'collecting_photos'` → сохранение фото в Storage → обновление статуса точки `requested → received` → если остались `pending` — запрос следующей, иначе `step = 'photos_complete'`
  - *Взять из старого воркфлоу:* паттерн хранения статусов точек (pending / requested / received) и логику перехода к следующей точке
- [ ] **Шаг 5 — Финальный анализ:** RAG-запрос к Supabase pgvector → GPT-4o (план + все фото + нормативные фрагменты) → JSON изменений с классификацией

## 11. Решённые вопросы

### Доступ и авторизация
Платного доступа нет. Бот проверяет `chat_id` пользователя по белому списку в Supabase (таблица `allowed_users`). Если `chat_id` отсутствует — бот отказывает в доступе и не продолжает диалог. Список пополняется администратором вручную через Supabase.

### Административный интерфейс
Не требуется на данном этапе.
